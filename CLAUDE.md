# CLAUDE.md

Guidance for Claude Code (and humans) working on this repo.

## What this is

`@abdallahaho/ccline` is a single-file bash statusline for [Claude Code](https://docs.anthropic.com/en/docs/claude-code). Install is `npx @abdallahaho/ccline@latest` — it copies one script to `~/.claude/statusline.sh` and wires a `statusLine` entry into `~/.claude/settings.json`.

Two design constraints drive every decision in this codebase:

1. **It runs on every Claude Code render.** Latency matters. Every subprocess spawn is a cost.
2. **It can be killed mid-render** if it exceeds Claude Code's statusline timeout. It must never leave persistent side effects behind (stale lock files, half-written caches, etc.).

## Repository layout

```
ccline/
├── bin/
│   ├── statusline.sh        # the script — everything users actually run
│   └── install.js           # npm-bin entrypoint; copies statusline.sh to ~/.claude/ and edits settings.json
├── .github/
│   └── workflows/publish.yml  # release-it on push to main
├── package.json             # @abdallahaho/ccline — npm package metadata + release-it config
├── README.md                # user-facing docs
├── how-it-works.md          # script internals — read this before touching statusline.sh
├── CHANGELOG.md             # auto-generated by @release-it/conventional-changelog
├── CLAUDE.md                # this file
└── LICENSE
```

`bin/statusline.sh` is the entire product. Don't factor it into multiple files — install needs to be a single `cp`, and shell sourcing across files adds fragility for no gain.

## Design principles

- **One script, no dependencies beyond the basics.** `bash`, `jq`, `curl`, `git` — nothing else. These are available everywhere Claude Code runs.
- **Single `jq` parse.** All input JSON fields are hydrated in one `eval` block (see `how-it-works.md`). Don't add per-field `jq` calls — each one is a subprocess.
- **Degrade quietly.** Missing tool, missing field, API failure — the statusline should produce *less* output, never an error message and never a crash.
- **Cross-platform by default.** Every `date` / `stat` call needs a BSD path and a GNU path. macOS ships BSD; Linux/WSL ships GNU; Git Bash is somewhere in between.
- **No persistent state.** The only thing written to disk is a 60s OAuth cache under `${TMPDIR:-/tmp}/claude-${UID}/`. No pidfiles, no session files.
- **Minimal visual noise.** Badges hidden when they'd duplicate the default (e.g. permission-mode badge only renders when non-default). Don't add segments that fire on every render unless they carry signal.

## Contributing

### Commit messages — conventional commits

The release workflow uses [`release-it`](https://github.com/release-it/release-it) with `@release-it/conventional-changelog` (angular preset). Commit type drives version bump:

| Type   | Effect                                                |
|--------|-------------------------------------------------------|
| `fix:` | patch bump (1.3.0 → 1.3.1) and appears under "Bug Fixes" |
| `feat:`| minor bump (1.3.0 → 1.4.0) and appears under "Features" |
| `feat!:` / `BREAKING CHANGE:` | major bump                     |
| `docs:` / `chore:` / `ci:` / `refactor:` | no release, but included in changelog |

Write the summary line in imperative mood and scope it with `(statusline)` when touching the script:

```
fix(statusline): avoid leaving stale .git/index.lock
feat(statusline): render xhigh and max effort levels
docs: document the effort ladder in the README
```

Avoid AI attribution or `Co-Authored-By: Claude` lines in commits — keep history clean.

### PR workflow

1. Branch from `main` — `fix/<slug>`, `feat/<slug>`, or `docs/<slug>`.
2. Keep PRs focused. One change per PR; don't bundle an unrelated refactor into a bug fix.
3. Open a PR against `main` via `gh pr create`. Include a test plan (even if manual).
4. On merge to `main`, GitHub Actions runs `release-it --ci`:
   - Bumps version per the conventional-commit types in the range
   - Updates `CHANGELOG.md`
   - Tags the release
   - Creates a GitHub release
   - Publishes to npm (OIDC trusted publishing — no npm token needed)

Only `bin/**` and `package.json` changes trigger the release workflow (see `.github/workflows/publish.yml` path filter), so docs-only PRs merge without cutting a release.

### Testing

There is no automated test suite — this is a bash script rendering ANSI, not a library. The two checks you should always run before pushing:

```bash
# Syntax check
bash -n bin/statusline.sh

# Smoke test with a minimal JSON payload
echo '{"model":{"display_name":"Opus 4.7 (1M context)"},"workspace":{"current_dir":"'"$PWD"'","project_dir":"'"$PWD"'"},"context_window":{"context_window_size":1000000,"used_percentage":12,"current_usage":{"input_tokens":1200,"cache_creation_input_tokens":0,"cache_read_input_tokens":0}}}' | bash bin/statusline.sh
```

For anything touching the render pipeline (new segment, new field, visual change), verify the interesting values render as expected by stubbing them into the JSON:

```bash
# Test rate-limit rendering
echo '{"rate_limits":{"five_hour":{"used_percentage":78,"resets_at":"2026-04-23T18:00:00Z"},"seven_day":{"used_percentage":92,"resets_at":"2026-04-28T12:00:00Z"}}, ...}' | bash bin/statusline.sh
```

For effort-level rendering, set `.effortLevel` in a throwaway settings file and point `$HOME` at a temp dir, or just edit `effort=` inline while testing.

### Cross-platform testing

If you don't have access to all target platforms, at minimum:

- **macOS**: primary dev target
- **Linux / WSL**: verify `stat -c` and `date -d` paths work
- **Git Bash**: verify OAuth token falls through to `~/.claude/.credentials.json` (since `security` and `secret-tool` are absent)

CI doesn't run platform matrix tests — reviewer eyeballs the BSD/GNU fallback pairs when reviewing.

## Common pitfalls

- **Don't call `jq` per field.** If you need a new input value, add it to the single `eval` block at the top of the script.
- **Don't assume `date -d` exists.** On BSD it doesn't — always use `to_epoch` / `format_reset_time` helpers that already handle both paths.
- **Don't add `git` commands without `GIT_OPTIONAL_LOCKS=0`** unless you've verified they can't take `.git/index.lock`. `git status` can take it during stat refresh; `git log`, `git rev-parse`, and `git config` are safe.
- **Don't add interactive prompts.** This runs non-interactively inside Claude Code — anything that reads stdin after the initial `cat` will hang.
- **Don't use bashisms that break in Git Bash.** Avoid `[[ =~ ]]` in hot paths (works, but slower and has MSYS edge cases); stick with POSIX `[ ]` and `case` where possible. Arrays and `${var:0:1}` slicing are fine.
- **Don't add segments that require a subprocess on every render** unless the signal is strong. The `claude --version` call was removed in 1.3.0 because it cost 300–500ms per cache miss.

## Release checklist (manual releases)

The workflow handles releases automatically, but if you need to trigger one manually:

```bash
# From main, after merging a PR:
gh workflow run publish.yml
# Or locally (requires npm + gh auth):
npm install -g release-it @release-it/conventional-changelog
release-it --ci
```
